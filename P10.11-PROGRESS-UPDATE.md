## Technical Details

### skipToContentLoop Structure

```lean
def skipToContentLoop (s : ScannerState) (fuel : Nat) : Except ScanError ScannerState :=
  match fuel with
  | 0 => .ok s
  | fuel' + 1 => do
    let mut s' := s
    -- ... whitespace/comment/newline handling ...
    match s'.peek? with
    | some c =>
      if isLineBreak c then
        s' := consumeNewline s'
        if !s'.isInFlowSequence then
          skipToContentLoop { s' with simpleKeyAllowed := true } fuel'
        else
          skipToContentLoop s' fuel'
      else .ok s'
    | none => .ok s'
termination_by fuel
```

**Key properties**:
1. Structurally recursive on `fuel` parameter
2. Base case: returns input state unchanged
3. Recursive case: processes one line, recurses with `fuel' = fuel - 1`
4. Termination proof automatic via `termination_by fuel`

**Why this enables proofs**:
- Induction on fuel provides natural proof structure
- No imperative mutation to reason about
- Each recursive call is clearly smaller (fuel decreases)

## Reflections ŌĆö Phase 2 Hindsight

### Unexpected Simplification 1: Reuse of Existing Helpers

**Expected**: Each inline loop would need a brand new helper function.

**Reality**: Two of the nine loops (scanNextToken trailing probes for `]` and `}`)
were identical to the existing `skipTrailingSpaces` helper that was already written
in structural recursion form. And a third (scanDocumentEnd) had the same logic but
couldn't reuse `skipTrailingSpaces` due to forward declaration order.

**Lesson**: When refactoring systematically, redundancy becomes visible. The imperative
style hid the fact that three loops were doing the same thing. Structural recursion
naturally factors common patterns into shared helpers.

### Unexpected Simplification 2: Batch Refactoring Is Faster Than Incremental

**Phase 1** (4 functions): ~6 hours across 3 sessions, with cascade discovery,
investigation of system limitations, and documentation.

**Phase 2** (9 functions): Single session, all 9 refactored, tested, and documented.

**Why**: Phase 1 taught the pattern. Phase 2 applied it mechanically. No cascades
(because there were no proof dependencies to discover), no system limitation
investigations (because we weren't proving theorems, just refactoring). The ROI
on learning the pattern was enormous.

**Also**: Reading all 9 loops at once revealed structural similarities. Loops 8-9
were obviously `skipTrailingSpaces`. Loops 4 and 8-9 were the same pattern.
Seeing them together made reuse obvious.

### Unexpected Challenge 1: Lean Doc Comments Are Position-Sensitive

**Bug encountered**: After inserting `skipDocEndWhitespace` between the docstring
for `scanDocumentEnd` and the `def scanDocumentEnd` declaration, Lean reported
"unexpected token '/--'".

**Root cause**: In Lean 4, `/-- ... -/` doc comments must be immediately followed
by the declaration they document. Inserting a different `def` between the docstring
and its target breaks the parser.

**Fix**: Move the helper definition *before* the docstring, not after it.

**Lesson**: Lean 4 doc comments are not free-floating ŌĆö they have implicit
attachment semantics. When inserting helper functions near documented definitions,
place them before the docstring block, not between the docstring and its `def`.

### Unexpected Challenge 2: Forward Declaration Order

**Bug encountered**: `scanDocumentEnd` (line ~1202) tried to call `skipTrailingSpaces`
(defined at line ~1368), causing "Unknown identifier" error.

**Root cause**: Lean 4 requires definitions to appear before their use (within a file).
Unlike Haskell, there is no mutual recursion block or forward declaration for
non-recursive definitions.

**Fix**: Created a dedicated `skipDocEndWhitespace` helper before `scanDocumentEnd`
instead of reusing the later `skipTrailingSpaces`. This is a case where DRY
was sacrificed for buildability ŌĆö but the functions are trivially small (8 lines each)
so the duplication cost is negligible.

**Alternative considered**: Move `skipTrailingSpaces` earlier in the file. Rejected
because it would require moving its callsite (`validateTrailingContent`) too,
creating a cascade of reorderings.

### Idiom: The Three Categories of Imperative Loops

Looking back across both phases, the 13 refactored loops fall into three categories:

**Category A: State-threading loops** (5 functions)
- `skipSpaces`, `skipWhitespace`, `skipToEndOfLine`, `hasTabInPrecedingWhitespace`, `skipToContent`
- Pattern: Thread `ScannerState` through recursive calls
- Proof impact: High ŌĆö these are called from critical paths

**Category B: Accumulator loops** (4 functions)
- `collectHexDigitsLoop`, `foldQuotedNewlinesLoop`, `parseBlockHeaderLoop`, `peekAt?Loop`
- Pattern: Thread both state and an accumulator (string, count, position)
- Proof impact: Medium ŌĆö affect scalar/escape proofs

**Category C: Bounded-iteration utilities** (4 functions)
- `advanceNLoop`, `skipDocEndWhitespace`, `skipTrailingSpaces` ├Ś 2 (reuse)
- Pattern: Simple advance-N-times or skip-while
- Proof impact: Low ŌĆö but eliminate `Id.run do` patterns entirely

**Insight**: Category A was the critical path (Phase 1). Categories B and C are
the long tail (Phase 2). The batch approach for B and C was much more efficient.

### Meta-Reflection: Infrastructure Debt Has a Long Tail

**Phase 1 narrative**: "We had 4 imperative loops blocking proofs. We refactored them.
Infrastructure debt: 0."

**Phase 2 reality**: "We had 9 more imperative loops that weren't blocking proofs *yet*.

**The distinction**: Phase 1 removed *blocking* debt. Phase 2 removed *latent* debt.
Both are real. Latent debt has a nasty property: it becomes blocking debt precisely
when you try to prove something new about the affected functions.

**Analogy**: Like fixing compiler warnings. They don't break the build today, but
they mask the signal from warnings that do matter, and they become errors when
compiler flags change.
