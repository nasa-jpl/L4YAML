## Attempted Proofs

### Attempt 1: emit_preserves_position_order

**Theorem**: Emitting a token preserves position monotonicity in the token array.

**Approach**: Case analysis on whether indices are in original array or are the new element.

**Result**: ❌ Failed due to dependent typing issues

**Technical Challenge**:
```lean
theorem emit_preserves_position_order (s : ScannerState)
    (h_ordered : ∀ (i j : Fin s.tokens.size), i.val < j.val → ...)
    : ∀ (i j : Fin (s.emit tok).tokens.size), i.val < j.val → ...
```

The problem: `Fin s.tokens.size` and `Fin (s.emit tok).tokens.size` are different types (since `(s.emit tok).tokens.size = s.tokens.size + 1`). When we try to use `h_ordered` on indices from the original array, Lean cannot automatically coerce the types.

**Error encountered**:
```
Type mismatch
  h_ordered ⟨↑i, hi⟩ ⟨↑j, hj⟩ hij
has type
  s.tokens[⟨↑i, hi⟩].pos.offset ≤ s.tokens[⟨↑j, hj⟩].pos.offset
but is expected to have type
  (s.tokens.push { pos := s.currentPos, val := tok })[i].pos.offset ≤
    (s.tokens.push { pos := s.currentPos, val := tok })[j].pos.offset
```

Even though these should be equal after applying `Array.getElem_push` lemmas, the dependent types prevent the substitution.

**What would be needed**:
- Custom coercion lemmas for `Fin` types
- More sophisticated use of `conv` mode to rewrite in specific positions
- Or: refactor to avoid dependent types (use `Nat` indices with bounds separately)

**Estimated effort**: 1-2 days of fighting the type system

## Root Cause Analysis

### Issue 1: Imperative For-Loops

The `scan` function uses Lean's imperative for-loop syntax:

```lean
def scan (input : String) := do
  let mut s := ScannerState.mk' input
  s := s.emit .streamStart
  for _ in [:fuel * 4] do
    match ← scanNextToken s with
    | some s' => s := s'
    | none => return final.tokens
  .error (.fuelExhausted ...)
```

This desugars to a complex recursive function using `ForIn` type class. The structure is approximately:

```lean
ForIn.forIn [:fuel * 4] initial_state (fun _ s =>
  match scanNextToken s with
  | some s' => ForInStep.yield s'
  | none => ForInStep.done (return ...)
)
```

**Challenges**:
1. The loop body has an early return (`return final.tokens`)
2. The loop uses mutable state (`s := s'`)
3. The loop bound is dynamic (`fuel * 4` depends on input)
4. Need to prove invariants hold after each iteration

**Standard Lean proof techniques don't work well**:
- `unfold`: Expands to unreadable recursive definitions
- `induction`: Would need manual induction on iteration count
- Loop invariants: No built-in support for imperative loops

### Issue 2: Array Dependent Typing

Arrays in Lean use dependent types for bounds checking:

```lean
structure Array (α : Type u) where
  data : List α

def Array.getElem (a : Array α) (i : Nat) (h : i < a.size) : α := ...
```

When we push an element, the size changes, creating a new type:
- Before: `Fin s.tokens.size`
- After: `Fin (s.tokens.size + 1)`

This makes it difficult to relate indices and elements before/after push operations.

### Issue 3: Early Returns in Loops

The `scan` function returns early from the loop:

```lean
for _ in [:fuel * 4] do
  match ← scanNextToken s with
  | none =>
    let final := unwindIndents s (-1)
    let final := final.emit .streamEnd
    return final.tokens  -- Early return!
```

This means the final array is constructed inside the loop body, not after the loop completes. Standard loop reasoning assumes the loop runs to completion and produces a final state.

## What Would Make This Provable

### Option 1: Refactor to Structural Recursion

Replace the imperative for-loop with explicit structural recursion:

```lean
def scanHelper (s : ScannerState) (fuel : Nat) : Except ScanError (Array (Positioned YamlToken)) :=
  match fuel with
  | 0 => .error (.fuelExhausted s.line s.col)
  | fuel' + 1 =>
    match scanNextToken s with
    | some s' => scanHelper s' fuel'
    | none =>
      let final := unwindIndents s (-1)
      let final := final.emit .streamEnd
      .ok final.tokens

def scan (input : String) : Except ScanError (Array (Positioned YamlToken)) :=
  let s := ScannerState.mk' input
  let s := s.emit .streamStart
  scanHelper s ((input.utf8ByteSize + 1) * 4)
```

**Advantages**:
- Direct induction on `fuel` parameter
- No implicit loop state
- Can prove properties by structural recursion

**Disadvantages**:
- Requires rewriting existing implementation
- May change performance characteristics
- Risk of introducing bugs
- Needs extensive re-testing

**Estimated effort**: 2-3 days refactoring + 3-5 days proving = 5-8 days total

### Option 2: Use Batteries Array Lemmas

Lean's Batteries library provides lemmas for array operations. Need to:

1. Import and use existing array lemmas
2. Develop custom coercion lemmas for `Fin` types
3. Use `conv` mode for targeted rewrites

**Example approach**:
```lean
theorem emit_preserves_position_order ... := by
  intro i j hij
  unfold ScannerState.emit
  conv_lhs => rw [Array.getElem_push]
  conv_rhs => rw [Array.getElem_push]
  split
  · -- Case: both in original
    ...
  · -- Case: j is new element
    ...
```

**Estimated effort**: 1-2 days per theorem, 6-8 days total

### Option 3: External Verification Tool

Use a different verification tool better suited for imperative code:

- **Why3**: Good for imperative programs with loops
- **Dafny**: Built-in support for loop invariants
- **Coq**: Has better tactics for imperative reasoning

Generate proof obligations and verify them externally, then trust the results.

**Estimated effort**: 5-10 days (learning new tool + verification)
