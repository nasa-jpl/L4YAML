-- We have: h_ordered : ∀ (i j : Fin s.tokens.size), ...

-- These are DIFFERENT TYPES in Lean's dependent type system:

## Root Causes

### 1. Imperative For-Loop Structure

The `scan` function (Scanner.lean:1968-1989) uses imperative syntax:

```lean
def scan (input : String) := do
  let mut s := ScannerState.mk' input
  s := s.emit .streamStart                    -- Line 1970
  for _ in [:fuel * 4] do                     -- Line 1976
    match ← scanNextToken s with
    | some s' => s := s'                      -- Mutable update
    | none => return final.tokens             -- Early return!
  .error (.fuelExhausted ...)
```

**Problems for formal verification**:
1. **Mutable state** (`s := s'`) — No direct induction principle
2. **Early return** — Result constructed inside loop, not after
3. **Dynamic fuel** — Loop bound depends on input
4. **Complex desugaring** — Expands to opaque `ForIn` recursion

**Why standard tactics fail**:
- `unfold`: Produces unreadable recursive definitions
- `induction`: No induction principle for imperative loops
- Loop invariants: Not natively supported for `do`-notation

### 2. Dependent Array Typing

Lean arrays use dependent types for safety:

```lean
def Array.getElem (a : Array α) (i : Nat) (h : i < a.size) : α
```

Pushing changes the type:
- Before push: `Fin n` where `n = s.tokens.size`
- After push: `Fin (n + 1)`

These are **different types**, not subtypes. Relating proofs across the push operation requires explicit coercions that Lean struggles to infer.

### 3. Early Returns in Loops

The `none` branch returns immediately with `final.tokens` (line 1988), bypassing the loop's natural completion. This breaks standard loop reasoning patterns where you prove:
1. Invariant holds initially
2. Each iteration preserves invariant
3. Therefore invariant holds after loop completes

Here, the "after loop" state may never exist — we return early!

## Paths to Completion

### Option A: Refactor to Structural Recursion (Recommended if full completion required)

**Approach**: Rewrite `scan` to use explicit recursion instead of for-loops:

```lean
def scanHelper (s : ScannerState) (fuel : Nat) :
    Except ScanError (Array (Positioned YamlToken)) :=
  match fuel with
  | 0 => .error (.fuelExhausted s.line s.col)
  | fuel' + 1 =>
    match scanNextToken s with
    | some s' => scanHelper s' fuel'
    | none =>
      let final := unwindIndents s (-1)
      let final := final.emit .streamEnd
      .ok final.tokens
```

**Advantages**:
- Direct induction principle on `fuel`
- Standard recursive proof techniques work
- No implicit loop state to reason about

**Disadvantages**:
- Requires rewriting production code
- Risk of introducing bugs
- Needs full test suite re-validation
- May change performance characteristics

**Estimated effort**: 7-12 days (2-3 refactor + 5-9 proving)
**Risk level**: Medium (code changes in critical path)
