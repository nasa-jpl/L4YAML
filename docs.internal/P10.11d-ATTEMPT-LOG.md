### Attempt 1: emit_preserves_position_order (Phase 1)

**Goal**: Prove that emitting a token preserves position monotonicity.

**Approach**: Case analysis on array indices (whether in original array or pointing to new element).

**Code**:
```lean
theorem emit_preserves_position_order (s : ScannerState) ... := by
  intro i j hij
  unfold ScannerState.emit
  by_cases hj : j.val < s.tokens.size
  case pos => -- Both i, j in original
    have hi : i.val < s.tokens.size := ...
    simp [Array.getElem_push, hi, hj]
    exact h_ordered ⟨i.val, hi⟩ ⟨j.val, hj⟩ hij
  case neg => -- j is new element
    ...
```

**Result**: ❌ **Failed**

**Error encountered**:
```
Type mismatch
  h_ordered ⟨↑i, hi⟩ ⟨↑j, hj⟩ hij
has type
  s.tokens[⟨↑i, hi⟩].pos.offset ≤ s.tokens[⟨↑j, hj⟩].pos.offset
but is expected to have type
  (s.tokens.push { pos := s.currentPos, val := tok })[i].pos.offset ≤
    (s.tokens.push { pos := s.currentPos, val := tok })[j].pos.offset
```

**Root cause**: Dependent typing issue. Even though `Array.getElem_push` should rewrite `(s.tokens.push x)[i]` to `s.tokens[i]` when `i < s.tokens.size`, Lean's type system can't automatically perform the coercion needed because:
- `i` has type `Fin (s.tokens.size + 1)`
- `h_ordered` expects indices of type `Fin s.tokens.size`

These are **different types** in Lean's dependent type system, not subtypes.

**What would be needed**:
- Explicit `Fin` coercion lemmas
- More sophisticated use of `conv` mode for targeted rewrites
- Or: rewrite to avoid dependent types (use `Nat` with separate bounds)

**Time spent**: ~1 hour

**Conclusion**: Dependent array typing makes this proof extremely difficult without refactoring the theorem statement.

### Attempt 2: scan_produces_at_least_two (Phase 2)

**Goal**: Prove that `scan` always produces at least 2 tokens (streamStart + streamEnd).

**Approach**: Use helper lemmas from ScannerProofs.lean to track token count through scan.

**Helper lemmas identified**:
```lean
-- From ScannerProofs.lean:240
theorem mk'_tokens_empty (input : String) :
  (ScannerState.mk' input).tokens = #[]

-- From ScannerProofs.lean:250
theorem emit_tokens_size (s : ScannerState) (tok : YamlToken) :
  (s.emit tok).tokens.size = s.tokens.size + 1
```

**Proof strategy**:
1. `mk' input` starts with 0 tokens (by `mk'_tokens_empty`)
2. After `emit .streamStart` (line 1970), we have 1 token (by `emit_tokens_size`)
3. Loop executes...
4. On success, `emit .streamEnd` (line 1987) gives ≥ 2 tokens (by `emit_tokens_size`)
5. Return at line 1988

**Code**:
```lean
theorem scan_produces_at_least_two ... := by
  unfold scan at h
  simp only [Bind.bind, Pure.pure] at h
  -- After simp, h contains complex ForIn operations
  sorry
```

**Result**: ❌ **Failed**

**Blocker**: The `scan` function uses `for _ in [:fuel * 4] do` which desugars to:
```lean
ForIn.forIn [:fuel * 4] initial_state (fun _ s =>
  match scanNextToken s with
  | some s' => ForInStep.yield s'
  | none => ForInStep.done (return ...)
)
```

**What happens when we unfold**:
1. `unfold scan` expands the function body
2. `simp` can partially reduce do-notation to `bind` operations
3. But then we're left with `ForIn` typeclass operations that don't reduce further
4. The `.ok` return happens inside the `none` branch (early return)
5. Standard tactics cannot reason about this control flow

**What would be needed**:
- Custom loop invariant tactic
- Lemma library for ForIn operations
- Case analysis lemma showing `.ok` only comes from line 1988
- Or: refactor to structural recursion

**Time spent**: ~2 hours

**Conclusion**: Even with the right helper lemmas, imperative loop reasoning is intractable with current Lean tactics.

## Key Findings

### Finding 1: Helper Lemmas Exist But Aren't Sufficient

The L4YAML codebase already has useful helper lemmas in `ScannerProofs.lean`:
- `mk'_tokens_empty` - Initial state has 0 tokens ✅
- `emit_tokens_size` - emit adds exactly 1 token ✅
- `emit_offset`, `emit_inputEnd`, etc. - Other properties ✅

However, these lemmas alone cannot overcome the fundamental barriers:
1. Dependent array typing (Attempt 1)
2. Imperative loop control flow (Attempt 2)

### Finding 2: Lean's Tactics Don't Support Imperative Loops Well

Standard Lean tactics struggle with:
- `for _ in [:n] do` loops with mutable state
- Early returns inside loops (Except.ok inside ForInStep.done)
- State mutations (`s := s'`)

The desugaring produces `ForIn` typeclass operations that:
- Don't have induction principles
- Can't be case-split effectively
- Resist both `unfold` and `simp`

### Finding 3: The Barrier Is Structural, Not Just Proof Engineering

We're not blocked because we haven't found the right tactic sequence. We're blocked because:

1. **For emit_preserves_position_order**: The theorem statement uses dependent types (`Fin`) that create incompatible types before/after push operations.

2. **For scan_produces_at_least_two**: The implementation uses imperative loops that don't have formal reasoning support in Lean's standard library.

These are **architectural issues**, not proof engineering challenges.

## Lessons Learned

### For Future Verification Projects

1. **Avoid imperative loops in verified code** - Use structural recursion instead
2. **Minimize dependent typing** - Prefer simpler types (Nat + bounds) over Fin
3. **Design for provability** - Consider verification during implementation
4. **Test tactics early** - Try proving simple properties before committing to an implementation style
