### Technical Challenge: Imperative Loop Reasoning

The remaining proofs face a fundamental difficulty: reasoning about Lean's
imperative `for`-loops. The `scan` function uses:

```lean
for _ in [:fuel * 4] do
  match ← scanNextToken s with
  | some s' => s := s'
  | none => ...
```

This is challenging because:
1. **Loop state mutation**: The `s := s'` updates are not directly visible in the type
2. **Fuel-bounded iteration**: The loop runs a fixed number of times, not until convergence
3. **Early exit**: The `none` case breaks the loop before fuel exhaustion

**Attempted approaches**:
- Unfolding the for-loop expands to complex recursive definitions
- Loop invariants in Lean require manual induction over iteration count
- The `do`-notation desugars to nested bind operations

**Current solution**: Extensive empirical validation via `#guard` checks demonstrates
the properties hold on all test cases. The theorem signatures establish the API
contract, even though the proofs use `sorry` internally.

**Future work**: Three possible approaches to complete the proofs:
1. Develop a loop invariant framework for `scanNextToken` iteration
2. Use a different proof technique (e.g., denotational semantics)
3. Refactor `scan` to use structural recursion instead of for-loops
