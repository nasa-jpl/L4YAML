### 3. Identified Technical Barrier 🚧

**Problem**: scan* functions use nested let bindings:

```lean
def scanFlowSequenceStart (s : ScannerState) : ScannerState :=
  let savedKey := s.simpleKey
  let s' := { s with simpleKey := { possible := false } }
  let s' := s'.emit .flowSequenceStart
  { s'.advance with flowLevel := s'.flowLevel + 1,
                    flowStack := s'.flowStack.push true,
                    simpleKeyStack := s'.simpleKeyStack.push savedKey }
```

**After unfold**, Lean produces:
```lean
have savedKey := s.simpleKey
have s' := { input := s.input, ..., tokens := s.tokens, ... }
have s' := s'.emit YamlToken.flowSequenceStart
have __src := s'.advance
{ input := __src.input, ..., tokens := __src.tokens, ... }
```

**Issue**: The tokens field is buried in nested structure updates. Tactics like `rw [emit_tokens_size]` can't find the pattern because the intermediate let bindings don't reduce.

**Attempts that failed**:
1. `rw [emit_tokens_size]` - pattern not found
2. `simp only [ScannerState.emit]` - no progress
3. `rfl` - not definitionally equal

**What would work**:
1. Manual tracking through each let binding (tedious, ~17 functions)
2. Refactoring scan* functions for explicit token operations
3. Stronger automation lemmas for field projections

## Architectural Insight ✅

**Key proven facts**:
1. ✅ `emit` only appends tokens (emit_tokens_size)
2. ✅ `advance` preserves tokens (advance_preserves_tokens)
3. ✅ `unwindIndents` adds tokens (unwindIndents_adds_tokens)
4. ✅ `saveSimpleKey` preserves tokens (saveSimpleKey_preserves_tokens)

**Architectural conclusion**: Every operation in scanNextToken either:
- Preserves tokens: skipToContent, advance, saveSimpleKey, field updates
- Adds tokens: unwindIndents, emit (in scan* functions)
- Never removes tokens

**Therefore**: `scanNextToken_adds_tokens` is architecturally sound.
**But**: Formal proof blocked by technical barriers.

## Lessons Learned

### Lesson 1: Definitional Equality Matters

**Finding**: Lean's definitional equality is strict. Nested let bindings and structure updates don't automatically reduce through field projections.

**Impact**: Even when we know architecturally that a property holds (emit adds, advance preserves), proving it formally requires matching Lean's reduction rules.

### Lesson 2: Code Structure Affects Provability

**Finding**: Just as imperative for-loops blocked verification, nested let bindings create barriers to formal proof.

**Precedent**:
- scan: for-loop → structural recursion = provable ✅
- unwindIndents: for-loop → structural recursion = provable ✅
- scan* functions: nested lets → ??? (current barrier)

**Generalization**: Verification-friendly code structure matters as much as correctness.
