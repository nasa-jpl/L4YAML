## Technical Barriers Identified

### Barrier 1: Imperative For-Loops

The `scan` function uses `for _ in [:fuel * 4] do` with mutable state and early returns. This desugars to complex `ForIn` recursion that resists standard proof techniques:

- `unfold` produces unreadable definitions
- No induction principle for imperative loops
- Early returns break standard loop reasoning patterns

**Solution**: Refactor to explicit structural recursion (7-12 days)

### Barrier 2: Dependent Array Typing

Lean's `Fin n` typing means array indices before and after `push` are **different types**. Proving properties across push operations requires explicit coercions that Lean struggles to infer.

**Example**:
```lean
-- Before: Fin s.tokens.size
-- After: Fin (s.tokens.size + 1)
-- These are incompatible types!
```

**Solution**: Use `Nat` indices with separate bounds, or sophisticated `conv` mode tactics

### Barrier 3: Foundation Dependency

Parser and end-to-end proofs depend on scanner proofs. Without completing the scanner foundation, the rest cannot proceed.
