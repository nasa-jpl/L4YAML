## 1. Unexpected Challenges

### 1.1 Do-Notation Opacity in Lean 4

**The Problem:**

When proving properties about functions written with `do` notation, the standard proof approach of `unfold` → `split` failed mysteriously:

```lean
def scanFlowEntry (s : ScannerState) : Except ScanError ScannerState := do
  if s.tokens.size > 0 then
    let lastTok := s.tokens[s.tokens.size - 1]!.val
    if lastTok == .flowSequenceStart || ... then
      throw (.invalidFlowEntry s.line s.col)
  let s_with_token := s.emit .flowEntry
  ...

-- Proof attempt:
theorem scanFlowEntry_adds_one_token (s : ScannerState) (s' : ScannerState)
    (h : scanFlowEntry s = .ok s') :
    s'.tokens.size = s.tokens.size + 1 := by
  unfold scanFlowEntry at h
  split at h  -- ❌ Error: Could not split an `if` or `match` expression
```

**The Root Cause:**

Do-notation in Lean 4 desugars to the *typeclass method* `Bind.bind`, not directly to `Except.bind`:

```lean
-- After unfold, the term contains:
have __do_jp := fun y => ...  -- join point
if condition then
  do
    let y ← throw error
    __do_jp y
else ...
```

The `split` tactic cannot see through `Bind.bind` to find the underlying `match` structure.

**The Solution:**

```lean
theorem scanFlowEntry_adds_one_token (s : ScannerState) (s' : ScannerState)
    (h : scanFlowEntry s = .ok s') :
    s'.tokens.size = s.tokens.size + 1 := by
  unfold scanFlowEntry at h
  simp only [bind, Except.bind] at h  -- ✅ Exposes the match structure!
  split at h <;> try contradiction
  injection h with h_eq; subst h_eq
  rw [emit_tokens_size]
```

**Key Insight:** `simp only [bind, Except.bind]` unfolds both the typeclass method and the concrete implementation, transforming opaque join points into matchable conditionals.

**Lesson:** This is now the **canonical two-step pattern** for proving properties of monadic functions:
1. `simp only [bind, Except.bind] at h`
2. `split at h`

This idiom should be documented in Lean4 style guides for verification projects.

### 1.2 Structural Recursion with Stateful Loops

**The Problem:**

Many scanner functions use fuel-bounded loops that advance state:

```lean
def collectAnchorNameLoop (s : ScannerState) (name : String) (fuel : Nat) : String × ScannerState :=
  match fuel with
  | 0 => (name, s)
  | fuel' + 1 =>
    match s.peek? with
    | some c =>
      if condition c then
        collectAnchorNameLoop s.advance (name.push c) fuel'
      else
        (name, s)
    | none => (name, s)
```

Proving token preservation requires showing `result.tokens = s.tokens`, but:
- The function calls `s.advance` (which preserves tokens)
- The recursive call changes both `s` and `acc`
- Standard induction doesn't immediately apply

**The Solution:**

```lean
theorem collectAnchorNameLoop_preserves_tokens (s : ScannerState) (acc : String) (fuel : Nat) :
    (collectAnchorNameLoop s acc fuel).snd.tokens = s.tokens := by
  induction fuel generalizing s acc with
  | zero =>
    unfold collectAnchorNameLoop
    rfl
  | succ fuel' ih =>
    unfold collectAnchorNameLoop
    split
    · split
      · -- Recursive case with s.advance
        rw [ih]  -- Apply IH to s.advance, not s!
        exact advance_preserves_tokens s  -- Bridge: s.advance.tokens = s.tokens
      · rfl
    · rfl
```

**Key Pattern:**
```lean
rw [ih]                           -- Apply IH to modified state
exact advance_preserves_tokens s  -- Connect modified back to original
```

**Lesson:** When proving loop invariants where state is modified at each iteration:
1. Make the induction `generalizing` over all changing parameters
2. Apply the IH to the *modified* state (not original)
3. Use composition lemmas to bridge from modified back to original

### 1.3 Structure Update Field Access

**The Problem:**

Scanner functions often return structures with only non-token fields modified:

```lean
def scanDocumentStart (s : ScannerState) : ScannerState :=
  ...
  { s_advanced with
    simpleKeyAllowed := true,
    allowDirectives := false,
    ... }  -- 5 fields updated, but NOT tokens

-- In proof:
have result : ScannerState := { s_advanced with ... }
-- Want: result.tokens = s_advanced.tokens
-- But: Lean doesn't auto-simplify structure projections through updates
```

The goal becomes:
```lean
({ s_advanced with simpleKeyAllowed := true, ... }).tokens.size ≥ ...
```

And `simp` doesn't reduce it automatically.

**The Solution:**

```lean
generalize h_result : { s_advanced with ... } = result
have h_tok : result.tokens = s_advanced.tokens := by
  cases h_result; rfl  -- ✅ Cases tactic reduces structure projection!
rw [h_tok]
```

**Why `cases` works:** It performs case analysis on the structure constructor, which forces reduction of the projection.

**Alternative (when structure is simple):**
```lean
dsimp only []  -- Sometimes sufficient to reduce projections
```

**Lesson:** Structure projections through `with` updates don't auto-reduce. Use `cases` on the generalized equation to force reduction.

**Lesson:** Use `private` for proof helper lemmas that compose into public theorems. Reserve `theorem` for user-facing correctness statements.

### 2.3 Pattern-Driven Development

**Discovery:**

After proving `collectAnchorNameLoop_preserves_tokens`, the pattern for *all* loop functions became clear:

```lean
theorem loop_preserves_property (s : ScannerState) (acc : α) (fuel : Nat) :
    property (loop s acc fuel) := by
  induction fuel generalizing s acc with
  | zero => unfold loop; <base case proof>
  | succ fuel' ih =>
    unfold loop
    split  -- on peek?
    · split  -- on condition
      · rw [ih]; <preservation lemma>
      · <return case>
    · <none case>
```

**Simplification:**

Instead of re-deriving each proof from scratch:
1. Copy the proven pattern
2. Replace function names
3. Adjust preservation lemmas
4. Done

**Time savings:** First proof took 30 minutes. Subsequent proofs took 5-10 minutes each.

**Lesson:** Once a proof pattern is established for a function class, document it explicitly. Use it as a template for similar proofs. Don't re-invent the wheel.

## 3. Key Idioms Discovered

### 3.1 The Monadic Proof Pattern

**Idiom:**
```lean
theorem monadic_function_property (s : ScannerState) (s' : ScannerState)
    (h : monadicFunction s = .ok s') :
    property s' := by
  unfold monadicFunction at h
  simp only [bind, Except.bind] at h
  split at h <;> try contradiction
  injection h with h_eq; subst h_eq
  <apply property lemmas>
```

**When to use:** Any function using `do` notation with `Except` monad.

**Components:**
1. `unfold` — expand definition
2. `simp only [bind, Except.bind]` — expose match structure
3. `split <;> try contradiction` — eliminate error branches
4. `injection; subst` — extract result value

**Frequency:** Used in 8/11 new helper lemmas.

---

### 3.2 The Composition Bridge Pattern

**Idiom:**
```lean
theorem function_adds_tokens (s : ScannerState) :
    (function s).tokens.size = s.tokens.size + n := by
  unfold function
  have h1 := helper_preserves_tokens ...
  have h2 := operation_adds_tokens ...
  simp only []
  rw [operation_lemma, h1, h2]
```

**When to use:** Function composes multiple operations (advance, emit, loop calls).

**Pattern:** Chain preservation and addition lemmas with `rw`.

**Example:** `scanAnchorOrAlias` composes:
- `advance` (preserves)
- `collectAnchorNameLoop` (preserves)
- `emitAt` (adds 1)

Result: `rw [emitAt_tokens_size, h_collect, h_adv]` — done.

---

### 3.3 The Structure Projection Reduction Pattern

**Idiom:**
```lean
generalize h_result : { s_modified with field := value, ... } = result
have h_proj : result.tokens = s_modified.tokens := by
  cases h_result; rfl
rw [h_proj]
```

**When to use:** Proving properties about structures returned with `with` updates.

**Why necessary:** Lean doesn't auto-reduce projections through `with`.

**Alternative (if simple):** `dsimp only []` may suffice.

---

### 3.4 The Split-Try-Contradiction Pattern

**Idiom:**
```lean
split at h <;> try contradiction
```

**When to use:** After exposing `match` structure in hypothesis `h : match ... with | .error => ... | .ok => ...`

**Effect:**
- Splits into one branch per pattern
- `.error` branches: `h : .error e = .ok s'` → `contradiction` closes goal
- `.ok` branch: hypothesis now has form `h : .ok ... = .ok s'`

**Saves:** 10-20 lines of boilerplate per proof.

---

### 3.5 The IH Application Pattern for Modified State

**Idiom:**
```lean
induction fuel generalizing s with
| succ fuel' ih =>
  split
  · -- recursive case with modified state
    rw [ih]  -- Apply IH to (modifiedState, ...)
    exact preservation_lemma  -- Bridge to original
```

**Key insight:** Apply IH to the *modified* state, then use preservation lemmas to connect back.

**Common mistake:** Trying to apply IH to original state directly.

### 8.2 Discovery: The 3-Case Split Pattern for peek?

**The Mistake:**

Initially approached peek? matches as 2-case splits:
```lean
split  -- on s.peek?
· -- some c case
  split  -- WRONG: this creates confusion
· -- none case
```

**The Correct Pattern:**

peek? always produces **3 cases** when split:
```lean
split  -- on s.peek?
· -- some <specific_char> (e.g., '>', '!', '\n')
· -- some c (generic character, c != specific)
· -- none
```

**Why This Matters:**

When the function definition uses:
```lean
match s.peek? with
| some '>' => ...
| some c => ...
| none => ...
```

The split tactic creates 3 branches corresponding to these cases. Trying to nest another split on the "some c" case leads to goal confusion and unnecessary complexity.

**Application:**

This pattern was critical for:
- `collectVerbatimTagLoop` (split on `'>'`)
- `collectAnchorNameLoop` (checking valid anchor characters)
- `collectTagHandleLoop` (split on `'!'`)

### 8.3 Discovery: Escape Sequence Proof Chains

**The Challenge:**

`processEscape` handles 12+ escape sequences:
- Simple: `\n`, `\t`, `\r`, `\\`, `\"`, etc.
- Hex: `\xXX` (2 hex digits)
- Unicode: `\uXXXX` (4 hex digits)

Each hex/unicode escape calls `parseHexEscape`, which calls `collectHexDigitsLoop`.

**The Solution: Build from Leaves to Root**

```lean
-- Level 1: Prove hex digit collection preserves tokens
theorem collectHexDigitsLoop_preserves_tokens (s : ScannerState) (hex : String) (n : Nat) :
    (collectHexDigitsLoop s hex n).snd.tokens = s.tokens := by
  induction n generalizing s hex with
  | zero => unfold collectHexDigitsLoop; rfl
  | succ n' ih =>
    unfold collectHexDigitsLoop
    cases h_peek : s.peek? with
    | none => simp []
    | some c =>
      simp []
      split
      · rw [ih, advance_preserves_tokens s]
      · rfl

-- Level 2: Prove hex parsing preserves tokens
theorem parseHexEscape_preserves_tokens (s : ScannerState) (n : Nat) (ch : Char) (s' : ScannerState)
    (h : parseHexEscape s n = .ok (ch, s')) :
    s'.tokens = s.tokens := by
  unfold parseHexEscape at h
  let (hex, s_after_hex) := collectHexDigitsLoop s "" n
  have h_collect := collectHexDigitsLoop_preserves_tokens s "" n
  ...

-- Level 3: Prove escape processing preserves tokens
theorem processEscape_preserves_tokens (s : ScannerState) (ch : Char) (s' : ScannerState)
    (h : processEscape s = .ok (ch, s')) :
    s'.tokens = s.tokens := by
  unfold processEscape at h
  simp only [] at h
  split at h <;> try contradiction  -- split on s.peek?
  repeat (split at h)  -- split on all escape character cases
  all_goals (
    first
    | (injection h with h_eq; cases h_eq; exact advance_preserves_tokens s)
    | (have h_adv := advance_preserves_tokens s
       have h_hex := parseHexEscape_preserves_tokens s.advance _ ch s' h
       rw [h_hex, h_adv])
    | contradiction
  )

-- Level 4: Use in main loop
theorem collectDoubleQuotedLoop_preserves_tokens ... := by
  have h_proc := processEscape_preserves_tokens s.advance ch s_after_escape heq
  ...
```

**Key Insight:**

Build proof chains bottom-up:
1. Prove lowest-level helper (`collectHexDigitsLoop`)
2. Prove each layer that depends on previous layer
3. Main proof uses top-level helper

This approach:
- Isolates complexity (each layer is simpler)
- Makes proofs reusable (processEscape used in double-quoted loop)
- Mirrors code structure (good proof hygiene)

### 8.4 Discovery: repeat + all_goals for Many Similar Cases

**The Problem:**

`processEscape` has a large match statement:
```lean
match s.peek? with
| some '0' => ... -- null
| some 'a' => ... -- bell
| some 'b' => ... -- backspace
| some 't' => ... -- tab
| some 'n' => ... -- newline
| some 'v' => ... -- vertical tab
| some 'f' => ... -- form feed
| some 'r' => ... -- carriage return
| some 'e' => ... -- escape
| some ' ' => ... -- space
| some '"' => ... -- quote
| some '/' => ... -- solidus
| some '\\' => ... -- backslash
| some 'x' => ... -- hex (2 digits)
| some 'u' => ... -- unicode (4 digits)
| ...
```

Proving each case individually would be tedious and repetitive.

**The Solution: repeat + all_goals + first**

```lean
theorem processEscape_preserves_tokens (s : ScannerState) (ch : Char) (s' : ScannerState)
    (h : processEscape s = .ok (ch, s')) :
    s'.tokens = s.tokens := by
  unfold processEscape at h
  simp only [] at h
  split at h <;> try contradiction  -- Error case
  repeat (split at h)  -- Split ALL escape cases at once
  all_goals (  -- Apply to ALL generated goals
    first
    | (injection h with h_eq; cases h_eq; exact advance_preserves_tokens s)
    | (have h_adv := advance_preserves_tokens s
       have h_hex := parseHexEscape_preserves_tokens s.advance _ ch s' h
       rw [h_hex, h_adv])
    | contradiction
  )
```

**How This Works:**

1. `repeat (split at h)` — Split on all conditional branches in h (creates 12+ goals)
2. `all_goals (...)` — Apply the same tactic block to all goals
3. `first | tac1 | tac2 | tac3` — Try tac1; if it fails, try tac2; if it fails, try tac3

**Result:**

Most cases: Simple escapes like `\n` → advance and return
- Pattern: `injection; cases; advance_preserves_tokens`

Hex/Unicode cases: Call parseHexEscape
- Pattern: Chain h_adv and h_hex lemmas

Error cases: Contradiction
- Pattern: `contradiction`

The `first` combinator tries each pattern in order, applying the first one that works.

**When to Use:**

- Functions with many similar branches (10+ cases)
- Each branch follows one of a few patterns
- Manually handling each case would be repetitive

**Time Savings:**

Without this pattern: 30+ lines (one per escape case)
With this pattern: 8 lines

### 8.5 Challenge: Nested Conditionals in collectPlainScalarLoop

**The Problem:**

The most complex proof was `collectPlainScalarLoop_preserves_tokens`. In the `!inFlow` + `isLineBreak` branch:

```lean
let s_after_newline := consumeNewline s
let (emptyCount, s_after_blanks) := skipBlankLinesLoop s_after_newline 0 bfuel inputEnd
let s_after_spaces := skipSpaces s_after_blanks
if s_after_spaces.col < contentIndent then
  .ok { content, spaces, state := s, terminated := true }
else if atDocumentBoundary s_after_spaces then
  .ok { content, spaces, state := s, terminated := true }
else
  -- Recursive call with folded content
  collectPlainScalarLoop s_after_spaces content' "" fuel' inFlow contentIndent inputEnd
```

**Initial Attempt (Failed):**

```lean
split at h  -- Split on skipBlankLinesLoop match
repeat (first | (split at h; try (injection h; rfl)) | (rw [ih, ...]))
```

This created unsolved goals because the `repeat` pattern didn't reach the nested conditionals.

**Working Solution:**

```lean
split at h  -- Split on col < contentIndent
· -- col < contentIndent case: returns terminated
  injection h with h_eq; cases h_eq; rfl
· -- col >= contentIndent: check atDocumentBoundary
  split at h
  · -- atDocumentBoundary = true: returns terminated
    injection h with h_eq; cases h_eq; rfl
  · -- atDocumentBoundary = false: recurse
    rw [ih _ _ _ h, h_sp, h_bl, h_cn]
```

**Key Insight:**

Nested if-then-else requires explicit split sequencing:
1. First split handles the first condition
2. In the else branch, second split handles the second condition
3. Each termination case: `injection; cases; rfl`
4. Recursive case: Apply IH with preservation lemmas

**Lesson:**

`repeat` and automation work well for **uniform branching** (like escape sequences).
For **sequential conditionals** with different outcomes, explicit split ordering is clearer and more maintainable.

### Idiom 7: The Proof Chain Pattern

**Idiom:**
```lean
-- 1. Prove leaf helper
theorem leaf_helper_preserves : ...

-- 2. Prove intermediate helper using leaf
theorem intermediate_helper_preserves : ... := by
  have h_leaf := leaf_helper_preserves ...
  ...

-- 3. Prove main function using intermediate
theorem main_function_preserves : ... := by
  have h_intermediate := intermediate_helper_preserves ...
  ...
```

**When to use:** Functions that call helpers that call other helpers (e.g., escape sequences).

**Benefit:** Each layer is simpler; proofs are reusable; mirrors code structure.

### Idiom 8: repeat + all_goals + first for Many Similar Cases

**Idiom:**
```lean
theorem many_cases_function (h : function s = .ok result) : property := by
  unfold function at h
  split at h <;> try contradiction  -- Error cases
  repeat (split at h)  -- Split ALL branches
  all_goals (
    first
    | (injection h with h_eq; cases h_eq; pattern1)
    | (have h1 := helper1; pattern2)
    | contradiction
  )
```

**When to use:** Functions with 10+ similar branches (escape sequences, character validation).

**Time savings:** Reduces 30+ lines to ~8 lines.

```lean
theorem nested_conditionals (h : function s = .ok result) : property := by
```

## 12. Function Decomposition for Proof Tractability

**Date:** 2026-03-08
**Focus:** Decompose `scanNextToken` (28 branches) and `scanDirective` (monolithic `do` block) into provable helpers
**Build:** ✅ 191/191 jobs, 869/869 tests
**Key Result:** Eliminated `set_option maxHeartbeats 6400000 in` from `scanDirective_monotonic` — now compiles with default heartbeats

### 12.1 Challenge: Simp Step Exhaustion in Large Dispatch Functions

**The Problem:**

`scanNextToken` had ~28 branch points — the single largest function in the scanner. When attempting to prove `scanNextToken_adds_tokens`, the standard approach:

```lean
theorem scanNextToken_adds_tokens (s : ScannerState) (s' : ScannerState)
    (h : scanNextToken s = .ok (some s')) :
    s'.tokens.size ≥ s.tokens.size := by
  unfold scanNextToken at h
  simp only [bind, Except.bind] at h
  split at h ...
```

hit a wall: the `simp only [bind, Except.bind]` step exhausted Lean's default 128K simp step limit. After unfolding `scanNextToken`, the term contained dozens of nested `Bind.bind` invocations. Each `simp` step had to traverse an exponentially growing term, and with ~28 branches the combinatorial explosion made the standard approach infeasible regardless of heartbeat budget.

**Root Cause Analysis:**

The cost isn't the number of goals after `split` — it's the `simp` preprocessing:
- With N bind points, `simp [bind, Except.bind]` must normalize a term of size O(N²) to O(N³)
- At N=28, this exceeds the 128K step limit
- Increasing `maxHeartbeats` would only delay the problem — it's fundamentally non-scalable

**Lesson:** Functions with more than ~7 branch points cannot be proven with the standard monadic proof pattern. The O(N²+) cost of `simp [bind, Except.bind]` requires structural decomposition, not just more heartbeats.

### 12.2 Solution: Decompose Functions Before Proving

**Strategy:**

Instead of fighting the proof, refactor the *implementation* to match proof constraints. Split large functions into focused helpers with ≤7 branches each, compose them in a thin wrapper, then prove each helper independently.

**scanNextToken decomposition (28 branches → 5 helpers + wrapper):**

| Helper | Responsibility | Branch Points |
|--------|---------------|---------------|
| `scanNextToken_preprocess` | skipToContent, unwindIndents, saveSimpleKey, check c | ~5 |
| `scanNextToken_dispatchStructural` | document markers, directives | ~6 |
| `scanNextToken_dispatchFlowIndicators` | `[`, `]`, `{`, `}`, `,` | ~5 |
| `scanNextToken_dispatchBlockIndicators` | `-`, `?`, `:` | ~3 |
| `scanNextToken_dispatchContent` | `&`, `*`, `!`, `\|`/`>`, `"`, `'`, plain | ~7 |
| `scanNextToken` (wrapper) | Compose with `match ← helper` | ~4 (match arms) |

Plus 3 pure Boolean helpers:
- `isBlockEntryCandidate` — checks `-` followed by blank in non-flow context
- `isKeyCandidate` — checks `?` at flow start or followed by blank
- `isValueCandidate` — checks `:` at flow start or followed by blank

**scanDirective decomposition (monolithic → 2 helpers + wrapper):**

| Helper | Responsibility | Branch Points |
|--------|---------------|---------------|
| `scanYamlDirective` | `%YAML` — parse version, validate, emit | ~6 |
| `scanTagDirective` | `%TAG` — parse handle + prefix, emit | ~2 (linear pipeline) |
| `scanDirective` (wrapper) | dispatch on directive name | 3 |

**Result:** Each piece is independently provable within default heartbeat limits.

### 12.3 Challenge: Bind-Free Wrapper Necessity

**The Problem:**

After decomposing `scanDirective`, the initial wrapper used `do` notation:

```lean
def scanDirective (s : ScannerState) : Except ScanError ScannerState := do
  if !s.allowDirectives then
    throw (.directiveAfterContent s.line)
  ...
  if name == "YAML" then scanYamlDirective s s_after_ws startPos
  else if name == "TAG" then scanTagDirective s s_after_ws startPos
  else .ok (skipToEndOfLine s_after_ws)
```

The `throw` desugars to `Bind.bind` (via join points), reintroducing the exact problem we were trying to solve — the hypothesis after `unfold` contained `have __do_jp := ...; Bind.bind ...`, and `split at h` couldn't see through the bind structure.

**The Solution:**

Remove `do` notation entirely. Replace `throw` with `.error`:

```lean
def scanDirective (s : ScannerState) : Except ScanError ScannerState :=
  if !s.allowDirectives then
    .error (.directiveAfterContent s.line)
  else
    let startPos := s.currentPos
    let s_after_percent := s.advance
    ...
    if name == "YAML" then scanYamlDirective s s_after_ws startPos
    else if name == "TAG" then scanTagDirective s s_after_ws startPos
    else .ok (skipToEndOfLine s_after_ws)
```

Now after `unfold scanDirective at h`, the hypothesis is pure `if-then-else` with `let` bindings — no `Bind.bind` anywhere. `dsimp only [] at h` reduces the `let` bindings, then `split at h` finds the `if` branches directly.

**Key Insight:** Wrapper functions for proof decomposition should **never** use `do` notation. Even a single `throw` introduces `Bind.bind`, which poisons the entire hypothesis. Use `.error` instead.

### 12.4 Challenge: `dsimp only []` Before `split` for Let/Have Reduction

**The Problem:**

After `unfold scanDirective at h`, the hypothesis contained:

```lean
h : (let startPos := s.currentPos;
     let s_after_percent := s.advance;
     let fuel := ...;
     let (name, s_after_name) := collectDirectiveNameLoop ...;
     let s_after_ws := skipWhitespace s_after_name;
     if name == "YAML" then ...) = .ok s'
```

The `split at h` tactic failed because it searches for top-level `if`/`match`, but the `let` bindings wrap the conditional. The tactic sees `let ... in ...` at the top level, not `if ... then ... else ...`.

**The Solution:**

```lean
unfold scanDirective at h
dsimp only [] at h    -- ← Reduces let/have bindings!
split at h            -- ← Now finds the `if` expression
```

`dsimp only []` performs definitional simplification with no rewrite rules, which is enough to beta-reduce `let` bindings and `have` statements, exposing the underlying conditional structure.

**When to use:** After `unfold` on any function that contains `let` or `have` bindings before its first `if`/`match`.

### 12.5 Simplification: Helper Theorem Composition

**Discovery:**

Once each helper has its own monotonicity theorem, the wrapper proof becomes a straightforward composition:

```lean
theorem scanDirective_monotonic (s : ScannerState) (s' : ScannerState)
    (h : scanDirective s = .ok s') :
    s'.tokens.size ≥ s.tokens.size := by
  unfold scanDirective at h
  dsimp only [] at h
  split at h
  · contradiction  -- !allowDirectives → .error
  · split at h
    · -- YAML
      have h_ws : ... .tokens = s.tokens := by rw [skipWhitespace_preserves_tokens, ...]
      exact scanYamlDirective_monotonic s _ _ s' h_ws h
    · split at h
      · -- TAG
        have h_ws : ... .tokens = s.tokens := by rw [skipWhitespace_preserves_tokens, ...]
        exact scanTagDirective_monotonic s _ _ s' h_ws h
      · -- unknown
        injection h with h_eq; subst h_eq
        rw [skipToEndOfLine_preserves_tokens, ...]; omega
```

**Pattern:** For each dispatch branch:
1. Establish token preservation hypothesis for intermediate state (`h_ws`)
2. Apply helper's monotonicity theorem via `exact`
3. Terminal branches: `injection; rw [preservation_lemmas]; omega`

**Result:** The old monolithic proof was 70 lines with `set_option maxHeartbeats 6400000`. The new composed proof is ~30 lines with **no heartbeat override at all**.

### 12.6 Simplification: Token Preservation Hypothesis Threading

**Discovery:**

Helper theorems need to know that intermediate processing (whitespace skipping, name collection) didn't change the token array. This is captured via a `h_ws` hypothesis:

```lean
theorem scanYamlDirective_monotonic (s : ScannerState) (s_after_ws : ScannerState)
    (startPos : YamlPos) (s' : ScannerState)
    (h_ws : s_after_ws.tokens = s.tokens)    -- ← threading hypothesis
    (h : scanYamlDirective s s_after_ws startPos = .ok s') :
    s'.tokens.size ≥ s.tokens.size := by
```

The caller constructs `h_ws` by chaining preservation lemmas:

```lean
have h_ws : (skipWhitespace (collectDirectiveNameLoop s.advance "" ...).2).tokens = s.tokens := by
  rw [skipWhitespace_preserves_tokens,
      collectDirectiveNameLoop_preserves_tokens,
      advance_preserves_tokens]
```

**Benefit:** Each helper theorem is self-contained — it doesn't need to unfold the pipeline that produced its input state. The composition happens at the call site.

## 13. Updated Idiom Catalog (Continued)

### Idiom 11: The Bind-Free Wrapper Pattern

**Idiom:**
```lean
-- BAD: do notation introduces Bind.bind
def wrapper (s : State) : Except Error Result := do
  if condition then throw error
  ...

-- GOOD: pure if-then-else with .error
def wrapper (s : State) : Except Error Result :=
  if condition then .error error
  else ...
```

**When to use:** Any wrapper function that exists primarily for proof decomposition.

**Why:** Even a single `throw` in `do` notation introduces `Bind.bind`, preventing `split` from finding conditionals. `.error` is a direct constructor — no bind overhead.

**Impact:** Eliminates the need for `simp only [bind, Except.bind]` entirely in the wrapper proof.

### Idiom 12: `dsimp only []` for Let/Have Reduction

**Idiom:**
```lean
unfold function at h
dsimp only [] at h       -- Reduce let/have bindings
split at h               -- Now finds if/match at top level
```

**When to use:** After `unfold` when the hypothesis starts with `let` or `have` bindings wrapping a conditional.

**Why:** `split` searches for top-level `if`/`match`. `let` bindings wrap the conditional, hiding it. `dsimp only []` beta-reduces without applying any rewrite rules.

**Alternative:** `simp only []` also works but may be slower. `dsimp` is preferred for pure definitional reduction.

**Idiom:**

```lean
  · -- Terminal branch: direct proof

```

**When to use:** Wrapper functions that dispatch to helper functions, where each helper has its own property theorem.

**Pattern:** Establish preconditions, then `exact` the helper's theorem.

## 14. Architecture: The Decomposition Design Principle

### 14.1 When to Decompose

**Rule of thumb:** If `simp only [bind, Except.bind]` times out, the function needs decomposition. This typically happens at ~10+ branch points.

More precisely:
- **≤ 7 branches:** Standard monadic proof pattern works directly
- **8–12 branches:** May need `maxHeartbeats` increase (fragile, non-portable)
- **≥ 13 branches:** Decomposition is necessary — `simp` step limit is structural

### 14.2 How to Decompose

1. **Group branches by semantic category** (structural, flow, block, content)
2. **Extract each group into a named helper** with a clear type signature
3. **Make the wrapper bind-free** (no `do`, use `.error` not `throw`)
4. **Compose with `match ← helper ...`** in the wrapper
5. **Prove each helper independently**, then compose in wrapper proof

### 14.3 Verification of Behavioral Equivalence

After decomposition, run the full test suite (869 tests) to verify behavioral equivalence. This is critical — refactoring for proof tractability must not change behavior.

### 16.2 Challenge: `scanBlockScalar` Heartbeat Exhaustion

The `scanBlockScalar_clears_simpleKey` and `scanBlockScalar_preserves_simpleKeyStack`
proofs require `set_option maxHeartbeats 400000` (3× the default 128,000).

**Root cause: exponential branching in `repeat (any_goals (split at h))`.**

`scanBlockScalar` is a 61-line function with **12 decision points** — 5 `match`
expressions, 4 `if` conditionals, and 3 monadic binds (`let ... ←`).  After
`unfold scanBlockScalar at h` and `simp only [bind, Except.bind, ...]`:

1. Each `split at h` doubles the proof state (one goal per branch).
2. `repeat (any_goals (split at h))` iterates until no more splits remain.
3. With 12 branch points, this creates up to $2^{12} = 4096$ goals before
   `all_goals (try contradiction)` prunes the impossible branches.

Compare with simpler functions that stay within the default budget:

| Function | Lines | Branch points | Goals before pruning | maxHeartbeats needed |
|----------|-------|---------------|---------------------|---------------------|
| `scanFlowEntry` | 9 | 3 | ~8 | default (128K) |
| `scanKey` | 20 | 4 | ~16 | default (128K) |
| `scanBlockScalar` | 61 | **12** | **~4096** | **400K** |

The heartbeat cost scales roughly as $O(2^b)$ where $b$ is the number of
branch points.  `scanBlockScalar` is at the threshold — 12 branches pushes
past the default budget, while simpler functions with 3–5 branches are
comfortably within it.

**Alternative approaches considered but not used:**
- **Decompose `scanBlockScalar`** (like was done for `scanDirective` in §12):
  Would eliminate the heartbeat override, but requires refactoring Scanner.lean
  and re-running all 869 tests.  The 3× override is mild enough that the
  cost/benefit doesn't justify the refactoring.
- **Manual case analysis** instead of `repeat split`: Would give finer control
  but produces verbose, fragile proofs.  The `repeat split + try contradiction`
  idiom is self-maintaining — it adapts when the function body changes.

### 16.3 Challenge: `rename_i` Binding Order After `repeat split`

After `repeat (any_goals (split at h))` + `all_goals (try contradiction)` +
`all_goals (try (simp only [Except.ok.injEq] at h; subst h))`, hypothesis names
are auto-generated (`heq✝`, `h✝¹`, `h✝`, etc.).  Using `rename_i` to recapture
them was unreliable because:

- `rename_i` binds names to inaccessible hypotheses **top-to-bottom** in
  the local context.
- After `repeat split`, the hypothesis order depends on which branches
  survived `try contradiction`, and how `subst` reordered the context.
- `rename_i h_poss h_eq` might bind `h_poss` to the character equality
  (`c = '"'`) rather than the `possible = true` condition.

**Solution:** Use `rename_i h_eq_dq _` to capture just the equation (known
to be the first inaccessible), then try both `scanDoubleQuoted_*` and
`scanSingleQuoted_*` lemmas via `first | ... | ...`.  The wildcard `_`
ignores the unreliably-ordered remaining hypotheses.

**Idiom 15: Partial `rename_i` with Fallback.**
When `rename_i` can't reliably capture all hypotheses, capture only the
one you need (first position is most stable) and use `first` to try
alternatives for the rest.

### 16.4 Challenge: `first | simp [...] | rw [...]` Silently Succeeds

In `all_goals first | simp [...] | rw [...]`, if `simp` closes *some* goals
but makes no progress on others, `first` considers `simp` successful (it
didn't throw an error) and never reaches the `rw` fallback.  The surviving
goals are left unsolved.

**Solution:** Replace `first | simp | rw` with two separate passes:
```lean
all_goals (try simp [...])
all_goals (rw [...])
```

The `try` wrapper makes the first pass non-fatal, letting it close what it
can.  The second pass applies `rw` to whatever remains.

**Idiom 16: Two-Pass Tactic Application.**
When `simp` and `rw` target overlapping but non-identical goal sets, use
`all_goals (try simp [...])` + `all_goals (rw [...])` instead of
`all_goals first | simp [...] | rw [...]`.

### 16.5 Challenge: `unfold f; split` vs. `simp only [f]; split`

For `scanTag`, the proof `unfold scanTag; simp only []; split` failed because
`unfold` normalizes `let` bindings (reducing them to their definitions),
which destroys the `match` structure that `split` needs.

**Solution:** `simp only [scanTag]; split` — this unfolds only the top-level
definition while preserving the internal `match` patterns.

**Idiom 17: Use `simp only [f]` Instead of `unfold f` for `split` Targets.**
When a function contains `let` bindings, `unfold` normalizes them away,
removing the `match` structure.  `simp only [f]` performs a single
delta-reduction without internal normalization, keeping `match` visible
for `split`.

### 16.6 Simplification: Specification Fix Unlocks Trivial Proof

The `parse_complete` theorem (EndToEndCorrectness.lean) had been marked
`sorry` across multiple sessions.  The obstacle was not proof difficulty
but a **specification mismatch**:

```lean
-- OLD (unprovable):
def ValidYaml (input : String) (docs : Array YamlDocument) : Prop :=
  ∃ (vts : ValidTokenStream),
    vts.input = input ∧
    Scanner.scan input = .ok vts.tokens ∧  -- ← unfiltered tokens!
    TokenParser.parseStream vts.tokens = .ok docs ∧ ...

-- NEW (provable):
def ValidYaml (input : String) (docs : Array YamlDocument) : Prop :=
  ∃ (filtered_tokens : Array (Positioned YamlToken))
    (raw_docs : Array YamlDocument),
    Scanner.scanFiltered input = .ok filtered_tokens ∧
    TokenParser.parseStream filtered_tokens = .ok raw_docs ∧
    docs = raw_docs.map YamlDocument.compose ∧ ...
```

The old definition used `Scanner.scan` (which includes `.placeholder` tokens)
while `parseYaml` calls `Scanner.scanFiltered` (which filters them out).
It also equated `docs` directly with `parseStream` output, but `parseYaml`
applies `YamlDocument.compose` (alias resolution + anchor stripping).

Once the definition matched the implementation pipeline (`scanFiltered` →
`parseStream` → `compose`), `parse_complete` became a 4-line proof:
```lean
have h_raw : parseYamlRaw input = .ok raw_docs := by
  unfold parseYamlRaw; simp only [h_scan, h_parse]
unfold parseYaml; rw [h_raw, h_compose]
```

**Lesson:** When a theorem resists proof for multiple sessions, check whether
the specification (not the proof) is wrong.  Specification bugs look like
proof difficulty.

### 16.7 Architecture: Bottom-Up Invariant Proof Structure

The full proof of `scanNextToken_maintains_simpleKeyAbove` was built in
four layers, each depending only on the layer below:

```

    ↓ uses
Layer 3: dispatch*_maintains_simpleKeyAbove      (4 lemmas, ~160 lines)
    ↓ uses
Layer 2: per-function simpleKey/Stack lemmas      (~17 pairs, ~500 lines)
    ↓ uses
Layer 1: loop-level preservation lemmas           (46 pairs, ~600 lines)
    ↓ uses
Layer 0: basic preservation (advance, emit, skip) (~12 pairs, ~150 lines)
```

Each layer compiles independently.  When a top-layer proof broke, the
error was always in the top layer — the bottom layers never regressed.
This contrasts with the earlier monolithic approaches that interleaved
concerns and made errors hard to localize.

## §18 — scanBlockScalar decomposition: eliminating maxHeartbeats

### 18.1 Problem

`scanBlockScalar` was a monolithic 61-line function with **12 branch points**
(5 `match`, 4 `if`, 3 monadic `←` binds). Proving properties required
`set_option maxHeartbeats 400000` (3× the default 128k) because
`repeat (any_goals (split at h))` generates O(2^b) intermediate goals.

Four proofs were affected:
- `scanBlockScalar_adds_one_token` — 90+ lines of manual case-splits
- `scanBlockScalar_preserves_prefix` — `maxHeartbeats 400000`
- `scanBlockScalar_clears_simpleKey` — `maxHeartbeats 400000`
- `scanBlockScalar_preserves_simpleKeyStack` — `maxHeartbeats 400000`

### 18.2 Solution: phase decomposition

Extracted three sub-functions, each with ≤5 branch points:

| Sub-function | Type | Branches | Purpose |
|-------------|------|----------|---------|
| `scanBlockScalarSkipComment` | `ScannerState → ScannerState` (pure) | 3 | Handle optional `#` comment after header |
| `scanBlockScalarConsumeNewline` | `ScannerState → Except ScanError ScannerState` | 3 | Consume required newline or EOF |
| `scanBlockScalarBody` | `... → Except ScanError ScannerState` | ≤5 | Indent detection, content collection, chomp/fold, emit, clear simpleKey |

The wrapper `scanBlockScalar` becomes:
```lean
def scanBlockScalar (s : ScannerState) : Except ScanError ScannerState :=
  let header := parseBlockHeaderLoop s.advance .clip none 2
  let s_after_comment := scanBlockScalarSkipComment (skipWhitespace header.2.2)
  match scanBlockScalarConsumeNewline s_after_comment with
  | .error e => .error e
  | .ok s_after_newline =>
    scanBlockScalarBody s s_after_newline header.1 header.2.1
      (s.peek? == some '|') s.currentPos
```

**1 branch point** in the wrapper (the `match` on the newline result).

### 18.3 Key technique: avoiding tuple destructuring in wrappers

The initial refactoring used:
```lean
  let (chomp, explicitOffset, s_after_header) := parseBlockHeaderLoop ...
```
This desugars to a `match` on the product type, creating an irrefutable
pattern match that `split` cannot see through because `have/let` bindings
wrap it. The fix: use `.1`/`.2` projections instead:
```lean
  let header := parseBlockHeaderLoop s.advance .clip none 2
  -- header.1 = chomp, header.2.1 = explicitOffset, header.2.2 = s_after_header
```
After `unfold; simp only [] at h`, the `let` bindings reduce via zeta-reduction,
exposing the inner `match` to `split at h`.

### 18.4 Proof structure: compose preservation lemmas

Each sub-function gets its own preservation lemmas:
```lean
-- Pure (no Except): direct equation
theorem scanBlockScalarSkipComment_preserves_tokens (s : ScannerState) :
    (scanBlockScalarSkipComment s).tokens = s.tokens

-- Except: conditional on success
theorem scanBlockScalarConsumeNewline_preserves_tokens (s s' : ScannerState)
    (h : scanBlockScalarConsumeNewline s = .ok s') : s'.tokens = s.tokens

-- Body: uses chain hypothesis
theorem scanBlockScalarBody_adds_one_token (s_orig s_nl : ScannerState) ...
    (h_tok : s_nl.tokens = s_orig.tokens)
    (h : scanBlockScalarBody ... = .ok s') :
    s'.tokens.size = s_orig.tokens.size + 1
```

The wrapper proofs compose them:
```lean
theorem scanBlockScalar_adds_one_token (s s' : ScannerState)
    (h : scanBlockScalar s = .ok s') :
    s'.tokens.size = s.tokens.size + 1 := by
  unfold scanBlockScalar at h
  simp only [] at h
  split at h
  · contradiction
  · exact scanBlockScalarBody_adds_one_token s _ _ _ _ _ s'
      (by rw [scanBlockScalarConsumeNewline_preserves_tokens _ _ (by assumption),
              scanBlockScalarSkipComment_preserves_tokens,
              skipWhitespace_preserves_tokens,
              parseBlockHeaderLoop_preserves_tokens,
              advance_preserves_tokens]) h
```

### 18.6 `dsimp only []` for let-in-if resolution

Inside `scanBlockScalarSkipComment`, the `let commentOk := match ... with ...`
binding followed by `if commentOk then ...` creates a `have`-wrapped `if`
that `split` cannot reach. Inserting `dsimp only []` performs zeta-reduction
(substituting the `let` value into the body), exposing the `if` for `split`.

Pattern:
```lean
  split    -- match s.peek?
  · split  -- match s.peekBack?
    · dsimp only []  -- reduce: let commentOk := expr; if commentOk → if expr
      split          -- if (isWhiteSpace c || ...)
      · exact skipToEndOfLine_preserves_tokens s
      · rfl
    · rfl
  · rfl
```
