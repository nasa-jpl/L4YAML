# P10.11a: Define ValidTokenStream Properties — Implementation Plan

## Context

The YAML parser verification project has a **specification gap**: `ValidTokenStream` defines what a valid token stream is, but no theorem proves that the scanner's `scan` function actually produces valid token streams. This is the missing link between the scanner implementation and the grammar specification.

**Current state:**
- ✅ `ValidTokenStream` structure defined in `Grammar.lean` with 4 properties:
  - `sizeGe2`: At least 2 tokens (streamStart + streamEnd)
  - `firstIsStreamStart`: First token is streamStart
  - `lastIsStreamEnd`: Last token is streamEnd
  - `positionsOrdered`: Token positions monotonically non-decreasing
- ✅ Infrastructure in `ScannerCorrectness.lean` with main theorem skeleton
- ✅ Envelope properties proven: `scan_produces_at_least_two`, `scan_first_is_streamStart`, `scan_last_is_streamEnd`
- ✅ Helper lemmas proven: token preservation for `emit`, `advance`, `unwindIndents`, and many scan* functions
- ✅ `scanValue` decomposed and fully proven (4 helper functions, no sorry)
- ❌ **4 theorems remain as `sorry`:**
  1. `scanNextToken_adds_tokens` — needs analysis of ~17 scan* branches
  2. `scanNextToken_preserves_prefix` — needs to show only appends
  3. `scanLoop_preserves_tokens` (recursive case) — depends on prefix preservation
  4. `scan_positions_ordered` — needs full loop invariant

**Why this matters:** Without these proofs, the claim that "the scanner conforms to the YAML specification" is unverified. The scanner state machine proofs (P10.10) show it maintains internal invariants, but don't establish that the output satisfies the grammar contract that the parser relies on.

## Implementation Plan

### Phase 1: Prove `scanNextToken_adds_tokens`

**File:** `Lean4Yaml/Proofs/ScannerCorrectness.lean:698-705`

**Approach:** Analyze the ~17 scan* functions called by `scanNextToken` and show each either returns an error or emits/appends tokens. The proof structure:

```lean
theorem scanNextToken_adds_tokens (s : ScannerState) (s' : ScannerState) :
    (scanNextToken s = .ok (some s')) →
    s'.tokens.size ≥ s.tokens.size := by
  intro h
  unfold scanNextToken at h
  -- Step through scanNextToken's structure:
  -- 1. skipToContent s — proven to preserve tokens
  -- 2. unwindIndents — proven to add tokens (unwindIndents_adds_tokens)
  -- 3. saveSimpleKey — preserves tokens
  -- 4. Match on character → call one of ~17 scan* functions

  -- Split on the if conditions and match branches
  split at h <;> try contradiction
  -- Each branch calls a scan* function
  -- Show each one adds tokens using existing lemmas or new helper lemmas
```

**Helper lemmas needed:** For each scan* function not yet proven, add a `scan*_adds_tokens` lemma. Many already exist (e.g., `scanFlowEntry_adds_one_token`, `scanBlockEntry_adds_tokens`, `scanValue_adds_tokens`). The remaining ones to prove:

- `scanDocumentStart_adds_one_token`: Calls `.emit .documentStart` → size + 1
- `scanDocumentEnd_adds_tokens`: Calls `.emit .documentEnd` + potential unwind → size ≥ 1
- `scanDirective_adds_one_token`: Emits version/tag directive
- `scanFlowSequenceStart_adds_one_token`: Calls `.emit .flowSequenceStart`
- `scanFlowSequenceEnd_adds_tokens`: Similar pattern
- `scanFlowMappingStart_adds_one_token`, `scanFlowMappingEnd_adds_tokens`
- `scanAnchorOrAlias_adds_one_token`, `scanTag_adds_one_token`
- `scanBlockScalar_adds_one_token`, `scanDoubleQuoted_adds_one_token`, `scanSingleQuoted_adds_one_token`
- `scanPlainScalar_adds_one_token`
- `scanKey_adds_one_token` — already proven!

**Pattern:** Each scan* function either:
- Directly calls `.emit` → use `emit_tokens_size`
- Calls `.emit` after other operations → chain with preservation lemmas
- Returns error → contradiction in success branch

### Phase 3: Complete `scanLoop_preserves_tokens` recursive case

**File:** `Lean4Yaml/Proofs/ScannerCorrectness.lean:717-780`

**Current status:** The base case (fuel = 0) and success path (scanNextToken = none) are proven. Only the recursive case (scanNextToken = some s') remains.

**Approach:**

```lean
theorem scanLoop_preserves_tokens (s : ScannerState) (fuel : Nat) (tokens : Array (Positioned YamlToken))
    (h : scanLoop s fuel = .ok tokens) :
    ∀ (i : Nat) (h_bound : i < s.tokens.size),
      ∃ (h_bound' : i < tokens.size), tokens[i] = s.tokens[i] := by
  -- ... existing proof for base and success cases ...
  | succ fuel' =>
    unfold scanLoop at h
    split at h
    · contradiction  -- error case
    · -- already proven: none case (success path)
    · -- NEW: recursive case (scanNextToken = some s')
      -- Apply inductive hypothesis on s' with fuel'
      have h_preserves : s'.tokens[i]'(...) = s.tokens[i] :=
        scanNextToken_preserves_prefix s s' ... i h_bound
      have h_ih : ∃ (h' : i < tokens.size), tokens[i] = s'.tokens[i] :=
        scanLoop_preserves_tokens s' fuel' tokens h i (...)
      -- Combine: tokens[i] = s'.tokens[i] = s.tokens[i]
      cases h_ih with
      | intro h' h_eq =>
        exists h'
        calc tokens[i] = s'.tokens[i] := h_eq
                      _ = s.tokens[i] := h_preserves
```

**Dependencies:** Requires `scanNextToken_preserves_prefix` from Phase 2.

**Estimated effort:** 1-2 hours (proof structure is already clear)

### Phase 4: Prove `scan_positions_ordered`

**File:** `Lean4Yaml/Proofs/ScannerCorrectness.lean:1035-1043`

**Approach:** Show that token positions are monotonically non-decreasing. This requires proving a loop invariant for `scanLoop`.

**Key insights:**
1. Initial state: After `emit .streamStart`, single token → ordering trivial
2. Loop invariant: At each iteration, if ordering holds for `s.tokens`, then ordering holds for `s'.tokens` after `scanNextToken`
3. Proof obligations:
   - Each new token has `pos.offset ≥` last token's offset
   - `emit_preserves_position_order` maintains ordering (already proven)
   - `advance_increases_offset` shows progress (already proven from `ScannerProgress.lean`)

```lean
theorem scan_positions_ordered (input : String) (tokens : Array (Positioned YamlToken))
    (h : scan input = .ok tokens) :
    ∀ (i j : Fin tokens.size), i.val < j.val →
      (tokens[i]).pos.offset ≤ (tokens[j]).pos.offset := by
  intro i j hij
  -- Unfold scan to expose scanLoop
  unfold scan at h
  -- Initial state has 1 token (streamStart) → ordering vacuous
  -- Apply loop invariant lemma:
  -- If s.tokens is ordered and scanNextToken s = .ok (some s'),
  -- then s'.tokens is ordered

  -- Use emit_preserves_position_order at each step
  -- Use advance_increases_offset for progress
  -- Induction on fuel in scanLoop
```

**Helper lemma needed:**

```lean
-- Loop invariant: scanNextToken preserves position ordering
private lemma scanNextToken_preserves_ordering (s : ScannerState) (s' : ScannerState)
    (h_next : scanNextToken s = .ok (some s'))
    (h_ordered : ∀ (i j : Fin s.tokens.size), i.val < j.val →
      (s.tokens[i]).pos.offset ≤ (s.tokens[j]).pos.offset) :
    ∀ (i j : Fin s'.tokens.size), i.val < j.val →
      (s'.tokens[i]).pos.offset ≤ (s'.tokens[j]).pos.offset := by
  -- Split cases:
  -- 1. Both i,j < s.tokens.size → use h_ordered + prefix preservation
  -- 2. i < s.tokens.size, j ≥ s.tokens.size → new tokens have offset ≥ s.offset
  --    which is > last old token (by advance_increases_offset)
  -- 3. Both i,j ≥ s.tokens.size → new tokens emitted in order
```

**Dependencies:** Requires `scanNextToken_adds_tokens` and `scanNextToken_preserves_prefix`.

**Estimated effort:** 4-5 hours (most complex proof, requires loop invariant reasoning)

- Phase 1 establishes that tokens are added (monotonicity)
- Phase 2 establishes that existing tokens are preserved (prefix preservation)
- Phase 3 composes Phases 1-2 to complete the recursive loop proof
- Phase 4 establishes position ordering using all previous results
- Phase 5 validates the complete proof chain

**Success criteria:**
- Zero `sorry` in `ScannerCorrectness.lean`
- `lake build` succeeds with 191/191 jobs
- `scan_produces_valid_tokens` fully proven (no axioms)
