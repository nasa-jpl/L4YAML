### 2. Refactored scanValue ✅

**Before** (heavily nested lets with complex logic):
```lean
def scanValue (s : ScannerState) : Except ScanError ScannerState := do
  let s := if ... then { s with ... } else s
  -- validation
  let s' := if s.simpleKey.possible then
    let s'' := s.insertAt ...
    if !s.inFlow then
      let s3 := s''.insertAt ...
      { s3 with ... }
    else { s'' with ... }
  else if ... then { s with ... }
  else ...
  let s'' := (s'.emit .value).advance
  -- validation
  .ok { s'' with ... }
```

**After** (explicit variable names):
```lean
def scanValue (s : ScannerState) : Except ScanError ScannerState := do
  let s_key_cleared := if ... then { s with ... } else s
  -- validation using s_key_cleared
  let s_prepared := if s_key_cleared.simpleKey.possible then
    let s_with_key := s_key_cleared.insertAt ...
    if !s_key_cleared.inFlow then
      let s_with_mapping := s_with_key.insertAt ...
      { s_with_mapping with ... }
    else { s_with_key with ... }
  else if ... then { s_key_cleared with ... }
  else ...
  let s_with_token := s_prepared.emit .value
  let s_after_advance := s_with_token.advance
  -- validation
  .ok { s_after_advance with ... }
```

**Token flow**:
1. `s` → `s_key_cleared` (simpleKey field only)
2. `s_key_cleared` → `s_prepared` (may add 0-2 tokens via insertAt)
3. `s_prepared` → `s_with_token` (adds .value)
4. `s_with_token` → `s_after_advance` (tokens unchanged)

## Key Insights

### Insight 1: Monadic Functions Are Harder

**Finding**: Functions using Except monad with error paths require different proof techniques than pure functions.

**Evidence**:
- Flow functions (pure): 3-line proofs ✅
- scanKey/scanValue (monadic): require handling do-notation

**Pattern for monadic functions**:
```lean
def scanFunction (s : State) : Except Error State := do
  let s1 := operation1 s
  if error_condition then throw error
  let s2 := operation2 s1
  .ok s2
```

Proof must handle:
- Split on error conditions
- Track through do-notation desugaring
- Handle multiple return paths

### Insight 2: insertAt Complicates Proofs

**Finding**: `insertAt` operations (used in scanValue) modify token array at arbitrary positions, making size tracking more complex.

**scanValue specifics**:
- May call `insertAt` 0, 1, or 2 times
- Each `insertAt` adds 1 token
- Then `emit` adds 1 more
- Result: 1-3 tokens added total

**Proof approach**: Would need lemmas about `insertAt` preserving/adding tokens.

### Insight 3: Architectural Understanding Sufficient

**Finding**: Even without complete formal proofs, the refactoring makes the token flow explicit and verifiable by inspection.

**Value delivered**:
- Clear variable names document the flow
- Each step's effect on tokens is obvious
- Tests validate functional correctness
- Foundation for future automation
