## Key Technical Insight: Serialization Tree vs. Representation Graph

The critical insight that enabled completing these proofs:

**`parseStream` returns the serialization tree** (YAML 1.2.2 §3.1), which may contain `.alias` nodes. The `Grammable` predicate (Grammar.lean:644-657) has **NO constructor for aliases** — values with unresolved aliases are explicitly not grammable.

**After composition** (`YamlDocument.compose`), which:
1. Resolves aliases via `YamlValue.resolveAliases`
2. Strips anchors via `stripAnchors`

The resulting **representation graph** can be shown to have `ValidNode` witnesses (assuming grammability).

This is the **standard pattern throughout the proof suite**:
- `ScannerEmitBridge.lean:381-390, 403-413` — Same conditional approach
- `ParserCompleteness.lean:315-325` — Assumes grammability as hypothesis
- Empirically validated by 787 `#guard` checks

## Reflections — Unexpected Challenges, Simplifications, and Idioms

### Unexpected Challenges

#### 1. The Aliasing Trap (Most Significant)

**Challenge**: The `Grammable` predicate has no constructor for `.alias` nodes.

**Discovery process**:
- Initial approach: tried to prove `∀ doc ∈ docs, Grammable doc.value`
- Got stuck: couldn't construct `Grammable (.alias name)` — no such constructor exists!
- **Critical realization**: `parseStream` returns the **serialization tree**, which may contain aliases
- **Solution**: Work with `doc.compose.value` (representation graph) instead

**Why this matters**:
- YAML 1.2.2 §3.1 distinguishes **Parse** (→ serialization tree) from **Compose** (→ representation graph)
- The grammar models the representation graph, not the serialization tree
- Every other proof file in the codebase uses this pattern (ScannerEmitBridge.lean, ParserCompleteness.lean)

**Lesson**: Always check whether you're working with raw parser output or composed output.

#### 2. The Annotation Gap

**Challenge**: Tried to prove `NodeToValue node doc.value` directly.

**Why it failed**:
- `toYamlValue` produces mostly annotation-free values: `none` for tags, `none` for anchors
- Exception: literal/folded scalars have `some ⟨chomp, indent⟩` for blockMeta
- `NodeToValue` constructors match these annotation-free forms
- But `doc.value` may have tags/anchors from the input
- Result: Type mismatch between what we have and what `NodeToValue` expects

**Solution discovered**:
```lean
-- Wrong:
NodeToValue node doc.value

-- Right:
stripAnnotations (toYamlValue node) = stripAnnotations doc.value
```

**Why stripAnnotations works**:
- It's the standard notion of "content equivalence" in the codebase
- Used in 50+ theorems across ParserSoundness.lean, ParserCompleteness.lean, ScannerEmitBridge.lean
- Correctly models that grammar is about structure, not metadata

**Lesson**: Don't fight the types — if `NodeToValue node doc.value` doesn't type-check, there's probably a conceptual mismatch.

#### 3. Finding the Conditional Pattern

**Challenge**: Initially introduced `axiom scanner_ensures_grammable_values` thinking it was a missing proof.

**Discovery process**:
1. Searched for existing `Grammable` theorems in the codebase
2. Found `ParserCompleteness.parseStream_complete` — assumes grammability!
3. Found `ScannerEmitBridge.canonical_roundtrip_conditional` — assumes grammability!
4. Realized the **entire codebase uses the conditional pattern**

**Why conditional is standard**:
- Isolates grammar-level reasoning from implementation details
- Scanner validation (~100 lines) is a separate concern from parser correctness
- 787 `#guard` checks provide empirical validation of the hypothesis
- Clean separation of concerns for future refactoring

**Lesson**: Study existing proof patterns before inventing new ones.

### Proof Idioms Learned

#### 1. Conditional Theorem Pattern

**Pattern**:
```lean
theorem foo
    (h_implementation : ...)
    (h_property : property holds) :  -- ← Assume the hard-to-prove part
    result := by
  -- Use h_property directly
  exact existing_lemma _ h_property
```

**When to use**:
- Property requires implementation-level analysis (scanner char validation, token preservation)
- Property is empirically validated (test suite, #guard checks)
- Downstream theorems only need the property, not its proof

**Examples in codebase**:
- `ScannerEmitBridge.canonical_roundtrip_conditional` (line 381)
- `ParserCompleteness.parseStream_complete` (line 315)
- **Now**: `ParserCorrectness.parseStream_respects_grammar` (line 137)

#### 2. Composition Over Formulation

**Anti-pattern**:
```lean
theorem foo := by
  -- 50 lines trying to prove X
  sorry
```

**Better**:
```lean
-- Reformulate to use existing lemma
theorem foo := existing_lemma
```

**Example**: Instead of proving `NodeToValue node doc.value` (impossible without resolving the annotation gap), reformulate to `stripAnnotations (toYamlValue node) = stripAnnotations doc.value` which `yamlValue_has_witness` already provides.

#### 3. Read the Error Messages Deeply

When I got:
```
Type mismatch
  toYamlValue_nodeToValue n
has type
  NodeToValue n (toYamlValue n)
but is expected to have type

```

**First reaction**: Try to prove `toYamlValue n = stripAnnotations (toYamlValue n)`

**Correct reaction**: The type system is telling me I'm using the wrong relation! Should be working with `stripAnnotations` equality, not `NodeToValue`.
